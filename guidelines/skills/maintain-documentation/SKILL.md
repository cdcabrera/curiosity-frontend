# Maintain documentation

## Sources of truth

- [CONTRIBUTING.md](../../../CONTRIBUTING.md) → **Development** → **Documentation** (JSDoc, lintfix, `build:docs`, tier-one `src` READMEs).
- [agent_testing.md](../../agent_testing.md) — spell and lint commands.

## Workflow

1. **JSDoc changes**
   - Emulate existing comment style in the same directory.
   - Avoid TS-only JSDoc that breaks plugins (see CONTRIBUTING: `[]`, `?` issues—prefer forms like `Array.<Type>`).
   - Autofix comment formatting: `npm run test:lintfix`.
2. **Tier-one README requirement**
   - Adding or **meaningfully changing** JSDoc on APIs may require updating `README.md` under the **first directory tier** of `src/` (e.g. `src/components/README.md`). Match sibling modules’ README style.
3. **Regenerate aggregated docs**
   - `npm run build:docs` (runs spell/lint then `docs:md` via `scripts/readme.docs.js` per `package.json`).
4. **Spelling**
   - Dictionary: `config/cspell.config.json`. Support spell for key markdown: `npm run test:spell-support` where applicable; locale/source spell: `npm run test:spell`.
5. **PR readiness**
   - CONTRIBUTING notes a **documentation lint** workflow (`.github/workflows/documentation_lint.yml`)—ensure README updates are included when JSDoc changes trigger that requirement.

## Do not

- Commit “ROUGH” autogenerated JSDoc from lintfix without human-readable completion (CONTRIBUTING warns outlines need further editing).
- Skip tier-one README updates when the documentation lint / team process expects them for your change.
